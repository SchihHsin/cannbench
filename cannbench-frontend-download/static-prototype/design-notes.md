# CANN-Bench 静态原型设计说明

这份说明只放设计取舍，不放在用户页面里。页面本身只保留用户完成任务需要看的信息和操作入口。

## 全站导航

位置：所有页面顶部导航。

改动：主导航保留“首页 / 任务包 / 排行榜 / 动态”，右侧只保留高频主动作“提交算子”；运行记录、语言、主题、退出登录和 API Token 收进账号下拉。

原因：“提交算子”是核心动作，需要随时可达；运行记录属于个人区查询动作，放进账号下拉可以减少顶部拥挤，同时仍然从首页和提交后的页面可达。

## 首页

位置：首页 hero、平台状态、第一次提交操作指南、选择起点、最近状态。

改动：首页先告诉用户完整路径：选任务包、看模板、提交算子、看运行记录；每一步都给一个界面小样、操作提示和直达按钮。保留 Level 入口和 runner / 队列 / 最近任务状态。顶部导航使用图标辅助识别，状态数据卡只保留用户能直接理解的数字和说明。

原因：第一次来的用户需要知道下一步做什么，不需要先理解平台管理视角。

## 任务包页

位置：`benchmarks.html` 的任务资源区、Level 区、任务列表。

改动：先展示 spec、模板、API、对应榜单，再按 Level 和具体算子进入提交。

原因：提交前最重要的是确认任务版本、模板结构、API 描述和榜单口径，避免用户上传后才发现任务包或硬件选错。

## 排行榜页

位置：`leaderboard.html` 的指标带、快速排名、读榜提示。

改动：把 Total Score、OPS、Cases、Hardware 放在表格前；表格里增加硬件、覆盖范围、最近提交和可操作入口。

原因：参考竞品后，排行榜要先让用户判断结果是否可比较，再看具体排名。不同硬件、不同任务包版本、不同覆盖范围不能混在一起理解。

## 个人工作区

位置：`account.html` 的提交算子、运行记录、API Token。

改动：提交表单和运行记录放在个人工作区内部；提交算子从顶部右侧直达，运行记录从账号下拉、首页状态区和提交后动作进入。

原因：提交是主动任务，入口要明显；运行记录是个人历史和排错信息，更适合放在账号相关入口里。

## 动态页

位置：`updates.html` 的规则、榜单、维护信息。

改动：动态页只保留任务版本、评测规则、runner 维护、平台发布等辅助信息。

原因：动态不是主流程入口，但会影响用户是否能提交、提交后是否能正常排队，以及某次结果是否仍然适用。

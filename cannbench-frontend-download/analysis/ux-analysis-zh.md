# CANN Bench 登录态中文站点 UX 分析

分析对象：`https://cannbench.com/` 登录态、中文界面、桌面视窗，以及下载到本地的线上生产前端包。

本次证据来自三部分：线上生产前端包中的路由/API/文案与组件逻辑、实际登录态桌面截图、页面文本抽取。桌面截图位于 `../evidence/user-screenshots/`，标注截图位于 `./annotated/`。管理页包含维护者数据，后续对外展示时建议脱敏。更完整的前端文件层面审计见 `./frontend-file-ux-audit-zh.md`。

## 1. 用户是谁

### 核心用户 1：算子提交者

这类用户通常是 CANN / Ascend NPU 算子开发者，或使用 Agent 自动生成 kernel 的参赛者。他来到网站的主要目标不是浏览品牌信息，而是完成一条闭环：

1. 知道这个 benchmark 评什么。
2. 找到任务规格和样例工程。
3. 按要求打包并提交 zip。
4. 看任务是否编译、精度、性能通过。
5. 在排行榜上理解自己的结果和排名。
6. 如有失败，下载产物或日志继续修复。

### 核心用户 2：结果观察者

这类用户可能是评测组织者、研究者、团队负责人或围观者。他的目标是判断哪个方案强、结果是否可信、不同硬件和任务子集如何比较。他最关注排行榜、评测集说明、版本动态和技术报告。

### 核心用户 3：维护者 / 管理员

维护者需要查看用户、任务、评测包、子集、权限和安全报告。管理后台属于高权限工作台，不应和普通用户主路径混在一起；它应更偏运维效率、风险提示、批量操作安全确认。

## 2. 用户主线任务

最重要的用户旅程是：

`了解规则 -> 选任务/下载规格 -> 准备提交包 -> 提交评测 -> 追踪任务 -> 理解失败/成功 -> 查看排行榜 -> 复盘与再次提交`

前端文件显示，当前站点已经覆盖了完整链路：评测集 bundle/reference/API description、zip 预览、CLI token、提交额度、job logs/artifacts、rerun、隐藏集、维护模式、分数刷新等能力都已经存在。主要问题不是功能少，而是每一步之间的“下一步”提示不够强，导致新用户需要自己推断：

- 概览页说清楚了平台是什么，但没有把“第一次提交”的路径压成清晰步骤。
- 评测集页展示了 Level 和子集，但下载 spec / examples / cases.yaml 的入口不够显性。
- 提交页有必要信息，也已有 zip 结构预览、模板说明、tag、聚合码、CLI 等高级能力；主要问题是“准备 zip -> 配置评测 -> 提交”的步骤关系不够强，新用户仍需要自己推断操作顺序。
- 任务页能看到状态，但失败后的诊断路径不明显。
- 排行榜数据完整，但指标解释和筛选层级较重，新用户不容易理解“总分、标准用例、隐藏用例、提交算子数”的关系。

## 5. 最大体验问题，基于前端文件复核

### 问题 0：功能已经很多，但没有按用户任务组织

生产包里的文案 key 分布很能说明问题：提交页相关 `sb.*` 约 100 个、任务详情 `jd.*` 约 80 个、排行榜 `lb.*` 约 60 个、管理 `ad.*` 超过 200 个。这说明产品能力很完整，但用户会面对大量功能入口和状态。体验优化的重点应是“把功能编排成流程”，而不是继续堆功能。

建议：

- 普通用户路径聚焦为：概览 -> 评测集 -> 提交 -> 任务 -> 排行榜。
- 维护者路径独立为：管理 -> 监控 / 用户配额 / 评测包 / 高风险维护 / 安全报告。
- 每个页面只突出当前阶段最重要的 1-2 个动作。

### 问题 1：提交页缺少“准备 zip 到提交”的步骤化引导

证据：`./annotated/05-submit-flow.png`

复核说明：之前把提交页截图中的窄视窗效果误判为表单布局问题。结合用户提供的完整桌面截图和前端包复核后，提交页是完整两栏布局；真正的问题不是宽度，而是“先下载/确认模板、再上传 zip、再配置评测、最后提交”的任务顺序不够显性。

建议：

- 保留两栏布局，但把表单组织成清晰步骤。
- 步骤顺序改为：选择评测集 -> 下载/确认模板 -> 上传 zip -> 选择硬件 -> 填标签/聚合码 -> 是否私有 -> 提交。
- 在上传区域附近增加“还没有 zip？”模板入口，并在提交前显示 zip 结构检查。

### 问题 2：新用户不知道“从哪里拿到正确 zip 模板”

提交页提示“基于 examples/aclnn_launch_example 或 direct_launch_example”，但这个路径对新用户而言仍是抽象的。评测集页也没有把“下载 spec、参考实现、cases.yaml”作为突出操作。

建议：

- 在评测集卡片上增加明确按钮：`下载任务包`、`查看提交模板`、`查看算子列表`。
- 在提交页上传框上方增加“还没有 zip？”区域，提供两个模板入口：`ACLNN Launch 模板`、`Direct Launch 模板`。
- 上传前提供 zip 结构校验预览：显示根目录是否包含 `build.sh`、`setup.py`、`CMakeLists.txt`、`cann_bench/`、`csrc/`、`cmake/`、`scripts/`。

### 问题 3：失败后的诊断闭环不够强

任务页能显示“编译失败、部分通过、成功”等状态，也有下载按钮，但用户下一步该看哪个日志、修哪个算子、是否重新提交，不够明确。

证据：`./annotated/06-jobs-feedback.png`

建议：

- 任务列表每行增加“失败摘要”列或展开区，例如：`3 个算子编译失败 / 12 个精度失败 / 查看首个错误`。
- Job 详情页应强化三段式结果：`构建`、`精度`、`性能`，每段显示通过数、失败数、首个错误、下载日志。
- 对失败状态给出下一步 CTA：`查看日志`、`下载产物`、`复制重提命令`。

### 问题 4：排行榜指标多，但解释成本高

排行榜有总分、平均分、基准加速比、理论加速比、提交算子、测例、标准/隐藏用例等指标。这些信息对资深用户有价值，但对第一次看的人负担较高。

建议：

- 顶部增加“排行榜怎么看”轻量解释，默认展开 3 条最关键口径：
  - 总分 = 标准用例 + 隐藏用例。
  - 同一团队聚合码取每个算子的历史最好成绩。
  - 提交算子数影响完整度，不能只看总分。
- 表格默认突出 `总分`、`提交算子`、`通过率/测例`、`最近更新`；高级指标放到展开列或详情页。
- 筛选区改为两层：常用筛选常驻，高级筛选折叠。

### 问题 5：顶部导航对不同角色没有分组

登录后顶部导航包含：概览、评测集、版本动态、排行榜、提交、任务、管理。对于普通参赛者来说，“提交”和“任务”是工作台；对于观察者来说，“评测集”和“排行榜”是主路径；对于维护者，“管理”是后台。

建议：

- 导航视觉上分为三组：
  - 了解：概览、评测集、版本动态
  - 参赛：提交、任务、排行榜
  - 维护：管理
- `管理` 仅对维护者显示，但仍建议做成右侧独立入口，避免普通主路径感知被后台功能打断。

## 6. 推荐的信息架构

### 中文版优先结构

顶部导航建议：

1. `概览`
2. `评测集`
3. `提交`
4. `任务`
5. `排行榜`
6. `版本动态`
7. `管理`，仅维护者可见

原因：用户的真实任务顺序是先理解任务，再提交，再追踪，再比较。当前把排行榜放在提交前面，更像展示导向；如果目标是提高参与和提交成功率，`提交` 与 `任务` 应更靠前。

### 首页首屏建议

首页不只做品牌介绍，应变成“任务入口页”：

- 主标题：`昇腾算子公开评测基准`
- 主 CTA：`开始一次提交`
- 次 CTA：`浏览评测集`、`查看排行榜`
- 新手路径卡片：
  - 1. 下载任务包
  - 2. 基于模板实现算子
  - 3. 上传 zip 提交
  - 4. 查看任务与排行榜
- 当前运行状态：`910C / 950PR 在线`，加上预计排队时间或最近任务状态。

## 7. 竞品参考：NVIDIA SOL-ExecBench

参考对象：`https://research.nvidia.com/benchmarks/sol-execbench`

本次按桌面整页截图重新采集了 5 个页面，均为 `1440px` 宽，且 metadata 显示无 cookie 弹窗、无横向溢出：

| --- | --- |
| 首页 | `../evidence/competitor-nvidia/01-home-skill-1440-fullpage-clean.png` |
| Problems | `../evidence/competitor-nvidia/02-problems-skill-1440-fullpage-clean.png` |
| Leaderboard | `../evidence/competitor-nvidia/03-leaderboard-skill-1440-fullpage-clean.png` |
| Status | `../evidence/competitor-nvidia/04-status-skill-1440-fullpage-clean.png` |
| Collection L1 | `../evidence/competitor-nvidia/05-collection-l1-skill-1440-fullpage-clean.png` |

### 值得 CANN Bench 借鉴的地方

1. **首页先讲清楚 benchmark 的价值，而不是先解释系统功能。** SOL-ExecBench 首屏直接表达“Real-World GPU Kernels Against Hardware Limits”，并把 `Explore Problems` 和 `View Leaderboard` 放成两个主入口。CANN Bench 首页也应先回答“这是一个什么评测、我下一步做什么”，再展示动态、报告和开源仓。

2. **Problems 页就是任务中心。** 竞品把 collections 放在顶部，下面是可搜索、可分页的 kernel 表格，并用 tags 标出 attention、normalization、rope、moe 等类型。CANN Bench 的评测集页也应从目录升级为任务包中心，把 `下载任务包`、`查看模板`、`查看 API`、`查看算子` 做成一级操作。

3. **Leaderboard 按 collection 和 kernel 两层组织。** 用户先看到 collection 排行入口，再进入具体 kernel leaderboard。CANN Bench 现在排行榜指标更复杂，因此更需要先给“快速看排名”的层，再把标准/隐藏用例、加速比、提交算子数放进深入分析层。

4. **Status 页提供公开实时反馈。** 竞品把提交 ID、时间、用户、问题、设备、状态做成 live feed，并显示自动刷新。CANN Bench 的任务页也有状态和日志能力，但更像列表；建议强化为“我的诊断工作台 + 全站状态流”的双层结构。

5. **Collection 详情把说明、题目、提交、排行榜放在同一页。** L1 页面里有 Description、Kernels、Submit Code、Leaderboard。CANN Bench 的评测集详情可以采用类似结构，但要强化 zip 模板、bundle、reference、api-description 和 cases.yaml，因为 CANN Bench 更偏工程交付。

### 不能照搬的地方

- SOL-ExecBench 更像公开 benchmark 门户，CANN Bench 是登录态评测工作台。CANN Bench 不能只做展示，要把 `提交 -> 任务 -> 失败诊断 -> 再提交` 做得更强。
- SOL-ExecBench 的核心概念是 SOL Score，口径相对单一；CANN Bench 有标准/隐藏用例、加速比、提交算子数、团队聚合码等多指标，需要更强解释。
- 竞品的 Status 是全站公开流；CANN Bench 还需要突出“我的任务”和失败修复路径，避免用户在全站任务里迷路。

## 8. 页面级改版建议

### 概览页

保留当前的科技感视觉，但减少首屏内部的重复信息。首屏应更快回答：

- 这是什么 benchmark？
- 我能提交什么？
- 我现在该点哪里？

建议加入“首次提交路径”横向步骤条，把 `浏览算子`、`下载模板`、`提交算子`、`看结果` 串起来。

### 评测集页

评测集页现在像目录。它应该更像“任务包中心”。

建议每个评测集卡片增加：

- `下载任务包`
- `查看 Level`
- `查看提交模板`
- `用于排行榜：是/否`
- `版本更新时间`

同时把 `Offical-Tasks` 拼写统一为 `Official-Tasks`，否则会影响专业感。

### 提交页

这是最高优先级页面之一。

建议改成分步式：

1. 选择评测集和目标硬件。
2. 下载模板或确认 zip 结构。
3. 上传 zip。
4. 填写公开标签和团队聚合码。
5. 私有提交设置。
6. 提交并跳转任务详情。

提交按钮禁用时要说明原因，例如 `请先上传 zip`、`请选择在线硬件`、`zip 超过 50 MB`。

### 任务页

任务页应从“任务列表”变成“诊断工作台”。

建议：

- 默认增加 `我的任务` 与 `全部任务` 切换。
- 失败任务优先展示可操作信息：失败阶段、首个错误、下载日志、重新提交。
- 对 `部分通过` 状态补充说明：哪些算子过了、哪些失败、是否计入排行榜。
- 删除按钮需要更弱化或加二次确认，避免和下载按钮同层级造成误操作风险。

### 排行榜

排行榜应支持两种模式：

- `快速看排名`：只看总分、提交算子数、硬件、用户、更新时间。
- `深入分析`：展开标准/隐藏、加速比、测例详情、任务来源。

建议把“更多筛选”默认折叠，指标解释做成固定帮助区或 hover tooltip。

### 账号页

账号页对 CLI 自动化用户很重要，但可以更强地连接提交路径：

- 在提交页提示“也可以用 CLI 提交”，链接到账号页对应区块。
- Token 生成后提供“复制并进入 CLI 教程”。
- CLI 脚本里的 `<CANN_BENCH_TOKEN>` 占位符需要更醒目，避免用户直接复制后失败。

### 管理页

管理页属于维护者工作台，应强调效率和安全：

- 高风险操作分区：用户权限、配额、删除任务、刷新分数。
- 表格增加批量筛选和操作历史。
- 对“安全报告”入口保留，但视觉上应和日常用户管理功能分开。

## 9. 中文优先，英文同步策略

先改中文版是合理的，因为当前主要问题不是翻译，而是信息架构和任务路径。建议流程：

1. 先确定中文版页面结构、按钮层级和状态文案。
2. 固化术语表：
   - 评测集 / Benchmark
   - 算子 / Operator
   - 任务 / Job
   - 提交 / Submission
   - 团队聚合码 / Team aggregation token
   - 标准用例 / Public cases
   - 隐藏用例 / Hidden cases
3. 英文版不要逐字翻译，要按英文开发者习惯表达。例如“提交算子”可译为 `Submit Kernel` 或 `Submit Operators`，不要机械使用 `Submit an operator` 覆盖全部场景。

## 10. 改版优先级

### P0：先修任务闭环

- 提交页改为清晰步骤流，并补充模板入口与 zip 结构检查。
- 任务页增加失败诊断和下一步操作。

### P1：降低理解成本

- 首页增加首次提交路径。
- 评测集页增加下载任务包/模板入口。
- 排行榜增加指标解释和快速/深入两种阅读层级。

### P2：完善高级用户和维护者体验

- 账号页和 CLI 工作流联动。
- 管理页做角色化分区、安全提示和操作审计。
- 英文版按中文结构同步。

## 11. 下一步设计产出建议

建议下一步直接进入中文版低保真改版，优先做 4 个页面：

1. 首页：从品牌展示改为任务入口。
2. 评测集：从目录改为任务包中心。
3. 提交：分步表单，强化模板入口和提交前校验。
4. 任务详情/任务列表：补齐失败诊断闭环。

完成中文版结构后，再统一做英文版文案和术语同步。
